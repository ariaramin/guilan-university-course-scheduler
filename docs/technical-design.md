# طرح فنی نسخه 0.9.2

تاریخ: ۱۴۰۵/۰۴/۲۷ (2026-07-18)

## جریان داده

`DOM زنده SADA → content.js → background.js → parser/normalizer/fingerprint → chrome.storage.local → dashboard.js → worker.js → engine.js`

- `dom-extractor.js` درخت iframeهای visible و هم‌دامنه را پیمایش و HTML table یا ARIA grid را به matrix تبدیل می‌کند. فقط وجود سرستون‌های «نام درس» و «برنامه زمانی» readiness جدول درس را تأیید می‌کند.
- Dashboard در بازشدن، بازگشت focus و درخواست دستی یک `requestId` تازه به service worker می‌دهد. worker ابتدا URL تب را محدود به host مجاز می‌کند، `PING_CONTENT_SCRIPT` می‌فرستد و فقط پس از پاسخ نسخه‌دار استخراج را آغاز می‌کند. اگر receiver وجود نداشته باشد، فایل‌های محلی را یک‌بار با `chrome.scripting.executeScript` تزریق و handshake را تکرار می‌کند.
- content script تا وجود ردیف معتبر، ناپدیدشدن loading indicator و ثابت‌ماندن fingerprint محتوای جدول در یک بازه کوتاه صبر می‌کند. timeout محدود است و snapshot ناپایدار منتشر نمی‌شود.
- پس از اولین خواندن، `MutationObserver` کوچک‌ترین container پایدار جدول/فرم را با debounce پایش می‌کند؛ load دوباره iframe و جایگزینی جدول observer را reconnect می‌کند.
- background فقط پاسخ همسان با آخرین `requestId` را parse/normalize می‌کند. fingerprint نهایی از فیلدهای پایدار هر درس ساخته می‌شود، بنابراین تغییر ظرفیت/استاد/زمان با تعداد ردیف ثابت نیز dataset تازه محسوب می‌شود.
- `cachedCourseDataset` شامل `schemaVersion`، آرایه دروس، timestamp عددی `extractedAt`، URL منبع، fingerprint، تعداد درس و ترم انتخابی است و پس از validation در یک write منتشر می‌شود. cache هنگام شروع current فرض نمی‌شود و فقط بعد از شکست واقعی live به‌عنوان fallback به کار می‌رود. داده دارای زمان معتبر پس از ۱۵ دقیقه stale است؛ نبود زمان به‌صراحت «نامشخص» اعلام می‌شود و به‌تنهایی به معنی قدیمی نیست. قالب قدیمی با زمان معتبر مهاجرت و قالب ناقص provisional می‌شود.
- تغییر صفحه‌بندی یا فیلتر خود SADA فقط ردیف‌های renderشده فعلی را می‌دهد؛ Dashboard این scope را با عبارت «جدول فعلی قابل‌مشاهده» اعلام می‌کند.
- `sada-parser.js` فقط جدولی با سرستون‌های شناخته‌شده را می‌پذیرد و در mismatch با خطای واضح fail closed می‌شود.
- `docx-xml.js` ZIP را با APIهای بومی مرورگر اعتبارسنجی، XML اصلی را استخراج و جدول Word را بازسازی می‌کند.
- `chart-parser.js` schemaهای متغیر و کنارهم را تشخیص می‌دهد، رکوردها را اعتبارسنجی/حذف تکرار و چارت را با درس‌های ارائه‌شده reconcile می‌کند.
- `engine.js` eligibility، تداخل کلاس/امتحان، bounded beam و رتبه‌بندی را مستقل از DOM اجرا می‌کند.
- `worker.js` engine را خارج از main thread اجرا می‌کند.
- `virtual-list.js` بازه visible، overscan و ردیف‌های گروه‌بندی‌شده را محاسبه می‌کند.

## استخراج چارت

قالب‌های پشتیبانی‌شده:

- DOCX: مسیر اصلی مستقیماً central directory فایل ZIP و `word/document.xml` را یک‌بار می‌خواند. `w:gridSpan`، `w:vMerge`، چند پاراگراف، جدول تو‌در‌تو، ردیف header تکراری، چند schema کنارهم و تغییر ترتیب ستون‌ها بازسازی می‌شوند. Mammoth فقط fallback برای سند معتبر ولی فاقد schema قابل‌تشخیص است.
- DOC: امضای باینری شناسایی می‌شود، اما به‌دلیل نبود parser مرورگری قابل اتکا با پیام تبدیل صریح به DOCX رد می‌شود.

PDF و قالب‌های spreadsheet/text عمداً در ورودی چارت پذیرفته نمی‌شوند. داده مبهم متن آزاد نیز پذیرفته نمی‌شود؛ فقط الگوی دارای واژه «واحد» یا جفت متن/عدد صریح وارد مرحله تطبیق می‌شود.

نرمال‌سازی شامل `ی/ي`، `ک/ك`، رقم فارسی/عربی/لاتین، کنترل‌های نامرئی و bidi، فاصله/نیم‌فاصله، نشانه‌گذاری و پرانتز، برچسب جنسیت، پیشوند «نام درس/درس»، پسوند گروه/کلاس/سکشن، جمع فارسی و واژه‌های عددی مستقل «یک…چهار» است. ردیف‌های تکراری با واحد متناقض یا کد مشترک و عنوان متفاوت `needs_review` می‌شوند. عنوان بخش، جمع و انتخاب کلی درس وارد رکورد درس نمی‌شود و دلیل رد فقط به‌صورت metadata ثبت می‌گردد.

ترتیب تطبیق:

1. انتخاب دستی ذخیره‌شده؛
2. عنوان خام دقیق و کد دقیق؛
3. نام دقیق نرمال‌شده؛
4. alias نرمال‌شده؛
5. Levenshtein similarity حداقل 0.92 با فاصله حداقل 0.04 از گزینه دوم.

تطبیق fuzzy فقط `probable_match` است و تا تأیید کاربر تعداد واحد را تغییر نمی‌دهد. وضعیت‌ها: `matched`، `probable_match`، `not_in_chart`، `unmatched_chart_item` و `needs_review`.

خروجی parser شامل تعداد جدول/ردیف، schema و confidence سربرگ، ادغام‌ها، رکوردهای ردشده/تکراری و خطاهای عددی است. رابط حداکثر ۱۰۰ ردیف بازبینی را هر بار وارد DOM می‌کند و اعمال داده تنها پس از تأیید کاربر انجام می‌شود.

## تولید برنامه با واحد نامشخص

درس بدون واحد معتبر در جدول باقی می‌ماند، با برچسب «تعداد واحد مشخص نیست» در بخش بازبینی دیده می‌شود و eligibility آن برای پیشنهاد رد می‌شود. کاربر می‌تواند درس متناظر چارت را انتخاب یا واحد را دستی ثبت کند؛ mapping در storage محلی ذخیره و تولید برنامه خودکار تکرار می‌شود. بنابراین هر برنامه خروجی فقط مجموع قطعی دارد و bucket یا برچسب «محاسبه جزئی» وجود ندارد.

رتبه‌بندی نهایی سه bucket دارد: دقیق، کمتر از هدف و بیشتر از هدف. داخل هر bucket فاصله، اولویت چارت، ترجیح کاربر، ظرفیت، روز حضور و gap بررسی می‌شود.

قواعد سخت حفظ می‌شوند: یک گروه از هر درس، عدم تداخل کلاس و امتحان، ظرفیت، passed، prerequisite/corequisite و گروه اجباری.

## Virtualization و UX

viewport ارتفاع ثابت دارد. `visibleRange` از `scrollTop / itemHeight` بازه را می‌سازد و ۷ ردیف overscan در هر طرف نگه می‌دارد. spacer ارتفاع کل را حفظ می‌کند و window با `translateY` جابه‌جا می‌شود. با ۲۰۰۰ درس، DOM همچنان کمتر از حدود ۴۰ ردیف دارد.

فهرست درس‌ها فقط یک جدول RTL با ردیف ۷۲px و header چسبان است. جدول حداقل عرض کنترل‌شده دارد و در viewport کوچک داخل همان ناحیه اسکرول افقی می‌شود؛ نمای کارت و state مربوط به آن حذف شده‌اند. جزئیات و کنترل‌های اجباری/ترجیحی/حذف در `dialog` باز می‌شوند تا هشت ستون اصلی جدول ثابت بمانند. رابط با tokenهای موجود، Vazirmatn محلی، focus قابل‌مشاهده، touch target موبایل و `prefers-reduced-motion` پیاده‌سازی شده است.

پیشنهادها در Worker ساخته می‌شوند و UI هنگام محاسبه نتیجه قبلی را حفظ می‌کند. فقط ۸ برنامه اول وارد DOM می‌شوند و ادامه با «نمایش بیشتر» اضافه می‌شود. مقایسه با یک دکمه `aria-pressed` روی هر برنامه انجام می‌شود، حداکثر دو برنامه می‌پذیرد و checkbox/ذخیره/انتخاب جداگانه ندارد. تغییر مقایسه فقط tray و همان کارت را به‌روزرسانی می‌کند و جدول درس‌ها دوباره render نمی‌شود.

نام و استاد هر گروه هنگام `rebuildGroups` یک‌بار نرمال و در `groupSearchIndex` ذخیره می‌شود؛ نتیجه فیلتر با کلید state memoize می‌شود. عضویت قطعی چارت نیز در `confirmedChartCourseIds` نگهداری می‌شود. فقط وضعیت `matched` با badge «در چارت شما» نمایش داده می‌شود؛ `probable_match` تا تأیید انسانی highlight نمی‌شود.

فیلتر جنسیت فقط metadata یک‌بار طبقه‌بندی‌شده را می‌خواند. نرمال‌سازی مشترک حروف عربی/فارسی، رقم‌ها، NBSP، نیم‌فاصله، zero-width/bidi mark، فاصله و شکل‌های مختلف پرانتز/نشانه‌گذاری را پوشش می‌دهد. عبارت‌های `فقط/مخصوص/ویژه/مختص` برای مرد/مردان/آقایان/برادران و زن/زنان/بانوان/خواهران به `male` یا `female` تبدیل می‌شوند؛ `all` و `unspecified` unrestricted هستند. اگر منبع هم‌زمان سیگنال مردانه و زنانه بدهد، رکورد `ambiguous`، دارای warning و در فیلتر خاص fail-closed است. عنوان خام و diagnostic نرمال‌شده حفظ می‌شود و عبارت محدودیت از عنوان نمایشی حذف می‌گردد.

## هویت بصری و design tokenها

فایل `extension/assets/university-of-guilan-logo.jpg` نسخه بدون تغییر و بایت‌به‌بایت یکسان دارایی رسمی ارائه‌شده توسط صاحب پروژه است (۱۴۶۷×۱۷۵۰، JPEG). لوگو برش، بازطراحی، recolor یا stretch نشده و در UI و چاپ با `object-fit: contain`، ابعاد ذاتی و سطح سفید استفاده می‌شود. full lockup عمودی برای آیکن ۱۶/۳۲px مرورگر مناسب نیست؛ تا زمان دریافت دارایی رسمی مربع یا emblem-only، manifest عمداً icon برند ندارد.

tokenهای افزوده یا اصلاح‌شده در `dashboard.css`:

- رنگ برند: `--color-brand-header`، `--color-brand-header-border`، `--color-brand-header-surface`، `--color-brand-header-text` و `--color-focus`؛
- اندازه‌ها: `--icon-lg`، `--control-height`، `--touch-target`، `--table-row-height`، `--table-min-width` و `--brand-logo-height`؛
- حرکت: `--duration-fast`، `--duration-standard` و `--easing-standard`؛ transitionهای موجود از همین tokenها استفاده می‌کنند؛
- چاپ: `--logo-width` برای اندازه ثابت و متناسب هویت در A4.

motion فقط برای ورود یکپارچه نتیجه/empty state با opacity و translate کوچک، feedback کنترل‌ها، sheet فیلتر و spinnerهای loading استفاده می‌شود. ردیف‌های جدول animate نمی‌شوند و `prefers-reduced-motion` همه حرکت‌های غیرضروری را حذف می‌کند. صفحه فیلتر موبایل هنگام بازبودن نقش modal می‌گیرد، focus را trap می‌کند، با Escape بسته می‌شود و focus را به کنترل آغازگر بازمی‌گرداند. skip link، توضیح screen-reader جدول، alt متن‌دار برای لوگوی معنایی و alt خالی برای تکرار تزئینی نیز اضافه شده‌اند.

## ذخیره‌سازی و امنیت

`chrome.storage.local` فقط dataset اتمیک گروه‌های استخراج‌شده، preferenceها، chart itemهای ساخت‌یافته، diagnostics شمارشی، manual matchها و اصلاح واحدها را نگه می‌دارد. raw chart file ذخیره نمی‌شود. حجم فایل ۲۰MB، پسوند/MIME/magic bytes، central directory، encryption flag، CRC، اندازه XML، وجود `word/document.xml`، واحد 0..10 و trust boundaryهای زمان/جلسه اعتبارسنجی می‌شوند.

وابستگی‌های runtime به‌صورت local vendor هستند:

- APIهای بومی `DecompressionStream` و `DOMParser`
- Mammoth 1.12.0 فقط برای fallback HTML
- Vazirmatn 5.2.8 (Arabic/Latin، وزن‌های 400/500/600/700)

افزونه مجوز عمومی `tabs` ندارد. علاوه بر `storage` و `scripting`، host permission فقط برای `https://sada.guilan.ac.ir/*` تعریف شده تا service worker بتواند به content script همان سامانه پیام بدهد یا همان فایل‌های بسته‌بندی‌شده را پس از نبود receiver تزریق کند؛ دسترسی cookie/token یا endpoint خارجی اضافه نشده است.

## پوشش تست و معیار عملکرد

`npm test` این مسیرها را پوشش می‌دهد: handshake و تزریق/شکست receiver، صفحه نامجاز و نبود تب، iframe/ARIA extraction، readiness هنگام تغییر تعداد ردیف، observer refresh، fingerprint با تعداد ردیف ثابت، رد پاسخ دیررس، schema و مهاجرت cache، parser SADA، گونه‌های Unicode محدودیت جنسیت و حالت متناقض، زمان و تاریخ فارسی، conflict و prerequisite، حذف واحد نامشخص از برنامه، ترتیب رتبه‌بندی، فایل واقعی DOCX الکترونیک، ZIP خراب/رمزدار، XML ناقص، header تکراری/کنارهم، ستون جابه‌جا، `gridSpan`/`vMerge`، جدول تو‌در‌تو، متن مخفی/چندخطی، واحد نظری+عملی، duplicate conflict، exact/normalized/alias/fuzzy matching، مدل چاپ فارسی، contract رابط و virtual range.

روی همین محیط توسعه:

- ۵۰۰ درس: حدود ۳۵۰ms
- ۲۰۰۰ درس: حدود ۱.۲–۱.۳s
- ۲۰۰۰ ردیف مجازی: کمتر از ۴۰ node ردیف در viewport تست

اعداد performance وابسته به سخت‌افزارند و تست‌ها budget محافظه‌کار ۳s/۵s دارند.

## محدودیت‌های واقعی باقی‌مانده

- فرمت باینری قدیمی DOC ورودی مستقیم ندارد و باید به DOCX تبدیل شود.
- متن ضروری داخل تصویر، text box و header/footer به‌عنوان رکورد درس خوانده نمی‌شود؛ وجود این بخش‌ها در diagnostics اعلام می‌شود و کاربر باید نسخه دارای جدول/متن عادی را بارگذاری کند.
- خروجی PDF با موتور چاپ بومی Chrome ساخته می‌شود؛ کاربر در گفت‌وگوی چاپ گزینه Save as PDF را انتخاب می‌کند تا شکل‌دهی RTL و صفحه‌بندی A4 توسط مرورگر انجام شود.
- قوانین رشته/ورودی فقط وقتی اعمال می‌شوند که منبع رسمی آن‌ها به مدل داده متصل باشد؛ افزونه آن‌ها را حدس نمی‌زند.
- تست نهایی روی session واقعی SADA باید توسط کاربر انجام شود، چون credential/session در scope پروژه نیست.
